RogueOne: Cryptographic Internals

DATE: 2026-04-19 | TAGS: cng secp256r1 aes-cbc-256 endianness

The RogueOne implant completely bypasses standard TLS/SSL implementations in favor of a raw, custom cryptographic protocol. By utilizing the native Windows Cryptography API: Next Generation (CNG) on the implant side and Python's cryptography module on the server side, the author minimized the binary footprint while maintaining strong forward secrecy.

This document details the exact byte-level mathematics and execution flow used to generate the shared keys and encapsulate the TLV traffic.

/////////////////////////////////////////////////////////////////////////////

Phase 1: ECDH P-256 Key Exchange

RogueOne relies on Elliptic Curve Diffie-Hellman (ECDH) over the NIST P-256 curve (also known as secp256r1). Because the initial connection is over a plaintext TCP socket, the keys must be negotiated dynamically without exposing the final session key to passive sniffers.

1. Generating Ephemeral Keys (Implant Side)

The implant invokes the Windows CNG API to generate an ephemeral key pair. It opens the BCRYPT_ECDH_P256_ALGORITHM provider and generates a 256-bit key pair using BCryptGenerateKeyPair and BCryptFinalizeKeyPair.

2. Wire Format & The Magic Prefix

To ensure cross-platform compatibility with the Python C2 server, the implant cannot send native Windows BCRYPT_ECCPUBLIC_BLOB structures over the wire. It must extract the raw mathematical points (X and Y coordinates) and construct a standard 65-byte uncompressed point.

// crypto.c: ExportPubkeyWire()
// The BCrypt blob starts with an 8-byte header, followed by 32 bytes of X, then 32 bytes of Y.
out[0] = 0x04; // Standard prefix denoting an "Uncompressed Point"
memcpy(out + 1,      blob + sizeof(BCRYPT_ECCKEY_BLOB),      32); // X Coordinate
memcpy(out + 1 + 32, blob + sizeof(BCRYPT_ECCKEY_BLOB) + 32, 32); // Y Coordinate

This 65-byte payload is transmitted via RawSendAll to the C2 server. The server receives it, verifies the 0x04 prefix, and parses it using EllipticCurvePublicKey.from_encoded_point(). The C2 server then generates its own 65-byte uncompressed point and sends it back to the implant.

/////////////////////////////////////////////////////////////////////////////

Phase 2: Secret Derivation & The Endianness Problem

Once both sides have exchanged public keys, they must derive the shared secret. In ECDH, the shared secret is the X-coordinate of the resulting mathematical point calculated by multiplying the local private key with the remote public key.

The Cross-Platform Endianness Quirk

This is where cross-platform cryptography often breaks. When the Python server calculates the shared secret (server_priv.exchange(ECDH(), implant_pub)), the resulting 32-byte X-coordinate is returned in Big-Endian byte order.

However, when the Windows CNG API (BCryptSecretAgreement followed by BCryptDeriveKey with BCRYPT_KDF_RAW_SECRET) calculates the exact same point, Microsoft returns the 32-byte X-coordinate in Little-Endian byte order. If both sides hashed these bytes directly, they would generate entirely different AES session keys.

The implant developer accounted for this by manually reversing the byte array in memory before hashing it:

// crypto.c: CryptoHandshake()
// BCryptDeriveKey returns the X coordinate in little-endian order on Windows.
// Reverse to get the canonical big-endian representation before hashing.
for (DWORD i = 0; i < secretLen / 2; i++)
{
    BYTE tmp = secretBuf[i];
    secretBuf[i] = secretBuf[secretLen - 1 - i];
    secretBuf[secretLen - 1 - i] = tmp;
}

Hashing into the AES Session Key

Directly using an ECDH shared secret as an encryption key is considered poor cryptographic practice. Both the implant and the server take the standardized 32-byte Big-Endian X-coordinate and pass it through SHA-256.

The resulting 32-byte digest is the final AES-256 session key, which is saved into the opaque CRYPTO_CTX struct on the implant, and passed into the Relay threads on the C2 server.

/////////////////////////////////////////////////////////////////////////////

Phase 3: The AES-256-CBC Engine

With the AES key established, all subsequent traffic is routed through CryptoSend and CryptoRecv. The underlying protocol remains TLV (Type-Length-Value), but the TLV chunks are completely encapsulated within an encrypted frame.

Frame Encapsulation Layout

Every single transmission on the wire looks exactly like this:

Offset Size Description
0x00 4 Bytes Plaintext Length: A 32-bit Little-Endian integer denoting the exact size of the decrypted payload.
0x04 16 Bytes Initialization Vector (IV): 16 bytes of cryptographically secure random data. Generated fresh for every frame to prevent CBC pattern analysis.
0x14 N Bytes Ciphertext: The AES-256-CBC encrypted payload, padded using PKCS#7 to the nearest 16-byte boundary.

Padding and State Handling

Because CBC mode operates on strict 16-byte blocks, payloads must be padded. The implant relies on the native BCRYPT_BLOCK_PADDING flag inside BCryptEncrypt to automatically apply PKCS#7 padding.

On the Python server side, the padding is implemented manually. The server calculates the necessary padding dynamically before encryption:

# server.py: crypto_send()
# PKCS#7 pad to next multiple of 16
pad_len = AES_BLOCK_LEN - (len(plaintext) % AES_BLOCK_LEN)
padded  = plaintext + bytes([pad_len]) * pad_len

cipher    = Cipher(algorithms.AES(key), modes.CBC(iv), backend=default_backend())

During decryption, the server reads the last byte of the decrypted block (pad_byte = padded[-1]), verifies it is between 1 and 16, and strips that exact number of bytes from the end of the buffer. Finally, it cross-references the length of the resulting plaintext with the 4-byte plaintext_len header sent at the beginning of the frame to ensure integrity.

Zero-Length Payloads

The C2 protocol utilizes empty packets (Length = 0) to signal the end of a fragmented TLV stream (an EOF marker). The cryptographic layer gracefully handles 0-byte encryptions. When plainLen is 0, the engine encrypts exactly one 16-byte block containing only PKCS#7 padding bytes (sixteen 0x10 bytes), ensuring the frame remains well-formed and perfectly aligned.
