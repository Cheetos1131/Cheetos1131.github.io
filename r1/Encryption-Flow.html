Cryptographic Internals: ECDH & AES-CBC

DATE: 2026-04-19 | TAGS: crypto math secp256r1 aes-cbc cng

The RogueOne framework completely abandons standard TLS/SSL overhead in favor of a raw, deeply integrated cryptographic protocol. By utilizing the native Windows Cryptography API: Next Generation (CNG) on the implant side and Python's cryptography module on the server side, the binary remains extremely small while mathematically ensuring forward secrecy.

This document details the complete mathematical foundation, API implementation, and data flow required to securely exchange keys and encapsulate traffic within this environment.

1. Elliptic Curve Diffie-Hellman (ECDH)

ECDH is an asymmetric key agreement protocol that allows two parties to establish a shared secret over an insecure channel. It relies on the algebraic structure of elliptic curves over finite fields, specifically the Elliptic Curve Discrete Logarithm Problem (ECDLP).

1.1 Mathematical Foundations

An elliptic curve over a finite prime field is defined by the Weierstrass equation:

y² = x³ + ax + b (mod p)

The core of ECDH is Scalar Multiplication, which involves adding a generator point to itself k times (denoted as k × G). While it is computationally trivial to calculate the resulting point Q, it is computationally infeasible to reverse the math and determine the scalar k (the private key) given only Q and G.

During the exchange:

  1. The Implant selects a private integer dA and computes its public key QA = dA × G.
  2. The C2 Server selects a private integer dB and computes its public key QB = dB × G.
  3. The parties exchange QA and QB over the plaintext TCP socket.
  4. Both parties calculate the shared secret. The Implant calculates S = dA × QB, and the Server calculates S = dB × QA.

1.2 The RogueOne ECDH Handshake Implementation

The implant explicitly utilizes the NIST P-256 curve (mathematically equivalent to secp256r1). The handshake is orchestrated by the CryptoHandshake() function.

Implant Key Generation & Export

The implant invokes the Windows CNG API to generate an ephemeral key pair by opening the BCRYPT_ECDH_P256_ALGORITHM provider and calling BCryptGenerateKeyPair.

To ensure cross-platform compatibility with the Python C2 server, the implant cannot send native Windows BCRYPT_ECCPUBLIC_BLOB structures over the wire. It must extract the raw mathematical points (X and Y coordinates) and construct a standard 65-byte uncompressed point.

// crypto.c: ExportPubkeyWire()
// The BCrypt blob starts with an 8-byte header, followed by 32 bytes of X, then 32 bytes of Y.
out[0] = 0x04; // Standard prefix denoting an "Uncompressed Point"
memcpy(out + 1,      blob + sizeof(BCRYPT_ECCKEY_BLOB),      32); // X Coordinate
memcpy(out + 1 + 32, blob + sizeof(BCRYPT_ECCKEY_BLOB) + 32, 32); // Y Coordinate

This 65-byte payload is transmitted via RawSendAll. The server receives it, verifies the 0x04 prefix, parses it using EllipticCurvePublicKey.from_encoded_point(), generates its own ephemeral keypair via generate_private_key(), and sends its 65-byte public key back to the implant.

2. The Endianness Problem & Key Derivation (KDF)

Using the raw X-coordinate of the ECDH shared secret directly as an AES key is a cryptographic anti-pattern, as it is not uniformly distributed. The raw secret must be passed through a Key Derivation Function (KDF)—specifically SHA-256—to "smash" the bits into a high-entropy key.

The Cross-Platform Endianness Quirk

When the Python server calculates the shared secret (server_priv.exchange(ECDH(), implant_pub)), the resulting 32-byte X-coordinate is returned in Big-Endian byte order.

However, when the Windows CNG API (BCryptSecretAgreement followed by BCryptDeriveKey with BCRYPT_KDF_RAW_SECRET) calculates the exact same point, Microsoft returns the 32-byte X-coordinate in Little-Endian byte order. If both sides hashed these bytes directly, they would generate entirely different AES session keys.

The implant developer accounted for this by manually reversing the byte array in memory before hashing it:

// crypto.c: CryptoHandshake()
// BCryptDeriveKey returns the X coordinate in little-endian order on Windows.
// Reverse to get the canonical big-endian representation before hashing.
for (DWORD i = 0; i < secretLen / 2; i++)
{
    BYTE tmp = secretBuf[i];
    secretBuf[i] = secretBuf[secretLen - 1 - i];
    secretBuf[secretLen - 1 - i] = tmp;
}

Both sides then pass the synchronized 32-byte Big-Endian X-coordinate into SHA-256. The resulting 32-byte digest becomes the final AES-256 session key, saved into the opaque CRYPTO_CTX struct on the implant.

3. Advanced Encryption Standard (AES) Core Mechanics

AES is a symmetric-key block cipher based on the Rijndael design, operating on fixed-size blocks of 16 bytes. Because RogueOne utilizes a 32-byte key derived from the ECDH handshake, it operates specifically as AES-256.

3.1 Key Expansion & Rounds

The 256-bit key cannot be used directly for every round. AES utilizes a Key Schedule to expand the 32-byte master key into a series of 16-byte Round Keys. AES-256 executes 14 identical rounds of mathematical transformations on a 4x4 matrix known as the State.

  • SubBytes: Every byte in the State matrix is independently replaced with another byte using a non-linear S-box.
  • ShiftRows: Provides diffusion by shifting the rows of the State matrix cyclically to the left.
  • MixColumns: Each column is multiplied against a fixed polynomial over Galois Field (2⁸), creating massive diffusion.
  • AddRoundKey: The 16-byte Round Key is combined with the State matrix using a bitwise XOR operation.

3.2 Cipher Block Chaining (CBC) Mode

By itself, the AES core algorithm only encrypts exactly 16 bytes deterministically (Electronic Codebook mode). To encrypt arbitrarily large TLV payloads securely, RogueOne deploys AES inside Cipher Block Chaining (CBC) mode.

In CBC mode, before a plaintext block is passed into the AES algorithm, it is XORed with the ciphertext of the previously encrypted block. For the very first block, an Initialization Vector (IV) is used.

IV Generation

To prevent pattern analysis, the IV must be unpredictable. The RogueOne implant generates a fresh, random 16-byte IV for every single frame transmitted using BCryptGenRandom with BCRYPT_USE_SYSTEM_PREFERRED_RNG. The Python server mirrors this behavior using os.urandom(AES_BLOCK_LEN).

3.3 PKCS#7 Padding

Because AES operates strictly on 16-byte blocks, payloads must be padded to a multiple of 16 bytes before encryption. RogueOne uses the PKCS#7 padding scheme. In PKCS#7, the value of each added padding byte is equal to the total number of bytes added (e.g., adding four bytes results in 0x04 0x04 0x04 0x04).

  • Implant implementation: Relies on the native BCRYPT_BLOCK_PADDING flag during BCryptEncrypt to automatically apply PKCS#7 padding.
  • Server implementation: The Python server calculates this mathematically before encryption (pad_len = AES_BLOCK_LEN - (len(plaintext) % AES_BLOCK_LEN)) and manually strips the bytes after decryption by reading the value of the final byte (pad_byte = padded[-1]).

4. Wire Frame Encapsulation

To orchestrate the transmission of these AES-CBC streams over TCP, RogueOne encapsulates all TLV traffic inside a rigid wire frame format. Every single chunk sent over the wire looks exactly like this:

Byte Offset Size Component Description
0x00 4 Bytes Plaintext Length A 32-bit Little-Endian integer denoting the exact unpadded length of the decrypted payload.
0x04 16 Bytes Initialization Vector The 16-byte random IV generated dynamically for the AES-CBC encryption of this specific frame.
0x14 N Bytes Ciphertext The AES-256-CBC encrypted payload, padded to a 16-byte multiple via PKCS#7.

Handling Zero-Length Payloads

The C2 protocol utilizes empty packets (Length = 0) to signal the end of a fragmented TLV stream (an EOF marker). The cryptographic layer gracefully handles 0-byte encryptions. When plainLen is 0, the engine encrypts exactly one 16-byte block containing only PKCS#7 padding bytes (sixteen 0x10 bytes). This ensures the frame structure remains cryptographically secure and perfectly aligned for the receiving parser.
