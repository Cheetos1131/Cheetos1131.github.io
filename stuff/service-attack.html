Threat Architecture Analysis

ROGUEONE Execution Flow

Educational analysis detailing the function relationships and execution sequence for User Account Control (UAC) bypass and Bring Your Own Vulnerable Driver (BYOVD) operations.

UAC Hijacking uac.c
Exploits auto-elevating Windows binaries via COM hijacking in the HKCU registry hive.
  • is_elevated() Checks the process token to confirm current privileges.
  • create_key() Writes the payload path and empty DelegateExecute to HKCU.
  • run_as_admin() Triggers slui.exe via ShellExecuteW.
  • bypass() Orchestrates the full UAC bypass sequence.
Service Management service.c
Interfaces with the Windows Service Control Manager to register kernel drivers.
  • service_exists() Queries the SCM to check if the driver service is already registered.
  • start_kernel_service() Calls StartServiceW to load the driver.
  • register_kernel_service() Uses CreateServiceW to register the .sys file as a SERVICE_KERNEL_DRIVER.
Vulnerable Driver driver.c
Manages the extraction and IOCTL communication with the signed vulnerable driver.
  • load() Writes embedded driver bytes to %TEMP%\MicrosoftUpdate11.01.sys.
  • init() Obtains a handle to \\.\IMFForceDelete123.
  • ForceDelete() Sends the 0x8016E000 IOCTL to forcefully delete files from Ring 0.
  • close_driver() Cleans up the driver handle.
01
UAC Bypass (Escalation to Admin)
Triggering auto-elevation via Registry COM Hijacking
1
bypass() → is_elevated() The sequence begins by checking the current process integrity level. If it is already Administrator, this stage is skipped entirely.
2
create_key() [DelegateExecute] Writes an empty string to the DelegateExecute value under HKCU\Software\Classes\Launcher.SystemSettings\Shell\Open\Command, forcing COM fallback.
3
create_key() [Default Value] Retrieves the current executable path and writes it to the (Default) value of the same registry key.
4
run_as_admin() Executes slui.exe. Because it is an auto-elevating binary, it reads the poisoned HKCU hive and launches the implant executable as High Integrity (Admin).
Transition The original low-privileged implant can now exit, as the newly spawned instance is running with full Administrator privileges, satisfying the prerequisites for Stage 2.
02
Driver Deployment & Registration
Dropping the payload and instructing the SCM to load it.
5
load() The elevated implant drops the embedded driver byte array to disk at %TEMP%\MicrosoftUpdate11.01.sys.
6
register_kernel_service() → service_exists() The implant queries the SCM to see if the disguised service name (MicrosoftUpdate11.01) is already registered.
7
CreateServiceW() If it does not exist, the implant calls CreateServiceW to register the newly dropped file as a kernel driver.
8
start_kernel_service() Executes the service, which loads the IMFForceDelete.sys driver into Ring 0. The OS allows this because the driver possesses a valid digital signature.
Defense Evasion Context By masquerading as "MicrosoftUpdate11.01", the service attempts to blend in with normal system services. The driver drops to %TEMP% to avoid triggering strict permissions in System32\drivers.
03
Exploitation (Ring 0 Execution)
Sending IOCTLs to perform arbitrary file deletion.
9
init() The implant uses CreateFileW to open a read/write handle to the driver's device interface: \\.\IMFForceDelete123.
10
ForceDelete() The implant formats targeted AV/EDR paths (prepending \??\) and sends a DeviceIoControl request using IOCTL 0x8016E000.
11
Kernel Deletion The vulnerable driver receives the IOCTL and deletes the specified files from the kernel context, completely bypassing user-mode file protections.
12
close_driver() Closes the handle to the driver device object once the destruction loop is complete.
Impact Security tools are physically removed from the disk while their services may still be running in memory, leading to an eventual crash or complete blinding of the endpoint's defensive capabilities.
