Threat Architecture Deep Dive

ROGUEONE Windows Implant

A comprehensive analysis of a modular C/C++ implant, detailing its execution flow from basic C2 beaconing to kernel-level AV destruction and shellcode injection.

Core Engine main.c command.c
The entry point of the implant. It uses NetworkInit to connect to the C2 server. It runs an infinite while(RecvMessage(...)) loop, passing structured TLV payloads to CommandDispatcher, which maps numeric IDs to functions like CmdRun or CmdPs.
C2 Protocol protocol.c network.c
Implements a custom Type-Length-Value (TLV) protocol over raw TCP sockets. It guarantees message boundaries using an 8-byte header: a 4-byte DWORD type (Command ID) and a 4-byte DWORD length, followed by the dynamically allocated byte payload.
Debug & Error debug.c error.c
Provides custom heap tracking via ImplantHeapAlloc and CheckHeapBalance to prevent memory leaks during prolonged execution. error.c maps custom implant error codes to human-readable wide strings.
01
System Reconnaissance & API Evasion
Discovering the host environment while avoiding user-land hooks.
system.c | process.c | module.c
1
Privilege Discovery Uses CheckTokenMembership to query if the implant holds an Administrator SID. This dictates whether the implant attempts UAC bypass or kernel driver loading.
2
Evasive Process Enumeration (CmdPsNt) Instead of standard APIs like CreateToolhelp32Snapshot, the implant resolves NtQuerySystemInformation from ntdll.dll dynamically to dump the SystemProcessInformation structure, bypassing EDRs that only hook high-level Win32 APIs.
3
PEB Module Walking (CmdModulesNt) To list loaded DLLs in a target process, it opens a handle, calls NtQueryInformationProcess to get the base of the PEB, and uses ReadProcessMemory to manually traverse the InLoadOrderModuleList linked list.
Attacker Objective By manually walking the Process Environment Block (PEB) and using undocumented NTAPIs, the malware operator can map out security products (by identifying their injected DLLs) without triggering behavioral alerts common to EnumProcessModules.
// Traversing the PEB_LDR_DATA NTSTATUS status = NtQueryInformationProcess(hProc, ProcessBasicInformation, &pbi...); ReadProcessMemory(hProc, peb.Ldr, &ldrData, sizeof(PEB_LDR_DATA), &bytes); LIST_ENTRY* nextEntry = ldrData.InLoadOrderModuleList.Flink;
02
Remote IAT Enumeration
Parsing the Import Address Table of running processes.
iat.c
1
Locating the Image Base Reads the PEB of a remote process to find the ImageBaseAddress (offset 0x10 in x64 architecture).
2
Parsing PE Headers Reads the remote IMAGE_DOS_HEADER, calculates the offset to the IMAGE_NT_HEADERS64, and extracts the Virtual Address (RVA) for the IMAGE_DIRECTORY_ENTRY_IMPORT.
3
Walking Import Descriptors Iterates through the remote IMAGE_IMPORT_DESCRIPTOR array. For each loaded DLL, it walks the IMAGE_THUNK_DATA64 array to extract either the imported function name or its ordinal number, streaming this back to the C2.
Attacker Objective IAT enumeration is a highly advanced recon technique. It allows an implant to statically analyze a running process in memory. Attackers use this to identify processes that import specific sensitive APIs (like crypto or credential APIs) or to find targets suitable for IAT hooking (replacing legitimate function pointers with malicious ones).
03
Process Hollowing & PPID Spoofing
Executing arbitrary shellcode inside a spoofed host process.
hollower.c
1
Parent Process Spoofing The implant enumerates processes to find explorer.exe. It uses InitializeProcThreadAttributeList and UpdateProcThreadAttribute to set Explorer as the parent, then creates notepad.exe in a CREATE_SUSPENDED state.
2
Dynamic Entry Point Resolution It queries the suspended process to find its base address, reads its PE header remotely, and calculates the AddressOfEntryPoint.
3
Injection & Execution Uses WriteProcessMemory to overwrite the legitimate entry point instructions with the raw shellcode sent over the C2 socket. Finally, ResumeThread is called, executing the payload inside the context of notepad.exe.
Defense Evasion Value PPID spoofing breaks process tree analysis (e.g., stopping analysts from seeing that the malicious implant spawned notepad). Process hollowing ensures the payload runs from a memory space backed by a Microsoft-signed executable, evading basic whitelisting.
// PPID Spoofing Setup UpdateProcThreadAttribute(si.lpAttributeList, 0, PROC_THREAD_ATTRIBUTE_PARENT_PROCESS, &hExplorer, sizeof(HANDLE), NULL, NULL); // Spawn Suspended CreateProcessA(NULL, "notepad", NULL, NULL, FALSE, EXTENDED_STARTUPINFO_PRESENT | CREATE_SUSPENDED, NULL, NULL, &si.StartupInfo, &pi);
04
UAC Bypass via COM Hijacking
Escalating from Standard User to High Integrity Administrator.
uac.c
1
Registry Poisoning The implant creates a registry key under HKCU\Software\Classes\Launcher.SystemSettings\Shell\Open\Command. It sets the default value to its own executable path and sets a DelegateExecute key to an empty string.
2
Trigger Auto-Elevation It calls ShellExecuteW on C:\Windows\System32\slui.exe. Because this binary is signed by Microsoft and its manifest requests auto-elevation, Windows grants it High Integrity without a UAC prompt.
3
Execution Hijack Upon launching, slui.exe queries the HKCU registry hive (which the attacker controls) to find its execution handler. It finds the poisoned path and executes the implant as an Administrator.
Why it works UAC is designed as a convenience boundary, not a hard security boundary. Auto-elevating binaries inherently trust the current user's registry hive (HKCU) over the system hive (HKLM) when resolving COM objects or shell commands.
05
BYOVD (Bring Your Own Vulnerable Driver)
Dropping a signed driver to forcibly blind or destroy AV/EDR.
driver.c
1
Driver Deployment If running as Admin, the implant writes an embedded driver array to %TEMP%\MicrosoftUpdate11.01.sys. This file is actually IMFForceDelete.sys, a legitimately signed component of IObit Malware Fighter.
2
Kernel Initialization Assuming the service was created, the implant opens a handle to the driver's device object: \\.\IMFForceDelete123.
3
Arbitrary File Deletion (Ring 0) The implant sends DeviceIoControl requests with the IOCTL 0x8016E000, passing target paths like \??\C:\Program Files\Windows Defender\MpCmdRun.exe. The driver executes the deletion from Ring 0, entirely bypassing user-mode file locks and EDR self-protection mechanisms.
Blind Spot Exploitation Many BYOVD attacks focus on patching kernel callbacks or terminating processes, which modern EDRs heavily monitor. Arbitrary file deletion from the kernel is highly destructive and often falls in a behavioral blind spot, causing the security product to silently fail on its next update or restart.
// Sending the Force Delete IOCTL BOOL result = DeviceIoControl( hDriver, 0x8016E000, // IObit Force Delete IOCTL (LPVOID)full_path, input_size, NULL, 0, &bytes_returned, NULL );
