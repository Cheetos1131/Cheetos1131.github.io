01
UAC Bypass via slui.exe
Standard User → High Integrity (Administrator)
T1548.002
a
Elevation check — Queries process token via OpenProcessToken + GetTokenInformation(TokenElevation). If already admin, skip this stage.
b
Registry hijack — Writes to HKCU\Software\Classes\Launcher.SystemSettings\Shell\Open\Command. Sets default value to VEN0m's own path, sets DelegateExecute to empty string to force COM fallback.
c
Trigger auto-elevation — Launches slui.exe via ShellExecuteW. Windows auto-elevates it (Microsoft-signed, in manifest). No UAC dialog shown.
d
Hijacked execution — Elevated slui.exe reads the poisoned registry key and re-launches VEN0m with admin privileges. Original process exits.
Why it works
Auto-elevating binaries trust HKCU registry paths that any standard user can write to. UAC was never designed as a hard security boundary.
02
Vulnerable Driver Loading
IMFForceDelete.sys • CVE-2025-26125 • IObit Malware Fighter
T1068
a
Extract embedded driver — VEN0m drops IMFForceDelete.sys (embedded in binary) to a temp directory. The driver is legitimately signed by IObit.
b
Create kernel service — Registers a kernel-type service with a disguised name (e.g. MicrosoftUpdate11.01). Includes checks for existing/stopped services.
c
Load into Ring 0 — Starting the service loads the driver into kernel space. Windows validates the signature and allows it — the driver is properly signed despite being vulnerable.
The irony
A driver component from an anti-malware product (IObit Malware Fighter) is being weaponized to defeat other security products. Driver signature enforcement only checks "who signed it" — not "is it safe."
03
AV/EDR Destruction
Kernel-level arbitrary file deletion
T1562.001
a
Open driver handle — VEN0m obtains a handle to the loaded IMFForceDelete.sys device object.
b
Send IOCTL for each target — Calls DeviceIoControl with IOCTL 0x8016E000, passing a wide Unicode string of the target path and its byte length.
c
Kernel-level force delete — The driver processes the request from Ring 0, deleting files even if locked by running processes. Bypasses all usermode file protections.
d
Gradual degradation — AV/EDR processes lose supporting DLLs, configs, and signature databases. They malfunction progressively and eventually stop protecting the system.
// Targeted directories
TARGETS = [
  "C:\Program Files (x86)\Kaspersky Lab",
  "C:\Program Files\Bitdefender",
  "C:\Program Files\Bitdefender Agent",
  "C:\Program Files\Windows Defender",
]
Novel approach
Previous BYOVD techniques killed processes (hookable) or patched kernel memory (scannable). File deletion is a blind spot — it doesn't trigger behavioral detections that monitor for process termination or memory tampering.
04
Persistence via Winlogon
Userinit registry key hijack
T1547.004
a
Drop copy — VEN0m copies itself to %LOCALAPPDATA% for a persistent file location.
b
Modify Userinit — Appends VEN0m's path to HKLM\...\Winlogon\Userinit (comma-separated from legitimate userinit.exe). Runs on every user login.
Detection opportunity
Modifications to the Winlogon Userinit key are almost always malicious. This is one of the easiest stages to detect if monitoring is in place.
Detection & Hunting Matrix
Stage 1 • UAC Bypass MEDIUM
Monitor registry writes under HKCU\Software\Classes\ for Shell\Open\Command modifications, especially DelegateExecute being set to empty.
Sysmon Event ID 13 • Registry Value Set
Stage 2 • Driver Load MEDIUM
Alert on new kernel services with driver files in temp directories. Hash-based detection for known vulnerable drivers. Monitor for sc.exe or service creation APIs.
Sysmon Event ID 6 • Driver Loaded
Stage 3 • File Deletion HARD
Rapid file deletions in security product directories from kernel context. File integrity monitoring on AV/EDR installations. DeviceIoControl calls to unusual device objects.
FIM alerts • AV health monitoring
Stage 4 • Persistence EASY
Winlogon Userinit modifications are rare and almost always malicious. Simple registry monitoring catches this reliably.
Sysmon Event ID 13 • HKLM Winlogon key
